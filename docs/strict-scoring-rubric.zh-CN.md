# AgentArena 严格评分细则

这份细则用于把 AgentArena 的分数从“测试通过即可高分”升级为“可解释、可审计、可复现”的工程质量评分。它参考了以下公开评测思路：

- SWE-bench 使用可复现环境评估补丁，并区分解决问题的 fail-to-pass 测试与防回归的 pass-to-pass 测试。
- SWE-bench Verified 强调从真实 GitHub issue 出发，判断编辑是否真正解决问题。
- EvalPlus 通过 HumanEval+、MBPP+ 的大量额外测试降低“刚好过样例”的偶然性，并加入效率评估。
- CodeBLEU 提醒代码评测不应只看文本相似度或单一准确率，还应考虑语法结构、数据流和语义。
- METR RE-Bench 强调真实任务环境、可执行评分函数、完整运行记录和对评测完整性的保护。

参考链接：

- SWE-bench harness: https://www.swebench.com/SWE-bench/reference/harness/
- OpenAI SWE-bench Verified: https://openai.com/index/introducing-swe-bench-verified/
- EvalPlus: https://github.com/evalplus/evalplus

## 总分结构

建议总分仍为 100 分，但不允许单靠公开测试通过直接拿满分。

| 维度 | 权重 |
| --- | ---: |
| 功能正确性 | 18 |
| 隐藏测试与回归保护 | 12 |
| 需求遵循与边界覆盖 | 10 |
| 代码质量与可读性 | 10 |
| 架构集成与最小侵入 | 10 |
| 安全与权限边界 | 8 |
| 性能与资源使用 | 8 |
| 测试质量 | 10 |
| 可维护性与长期演进 | 7 |
| 交付过程与可审计性 | 7 |

## 硬性门槛

以下情况触发封顶或直接判低分：

- 公开测试失败：最高 50 分。
- 隐藏关键测试失败：最高 60 分。
- 未产生有效代码或文档改动：最高 30 分。
- 修改评分脚本、测试断言、任务定义来绕过评测：最高 20 分。
- 删除或跳过关键测试：最高 20 分。
- 引入高危安全问题：最高 40 分，严重时 0 分。
- 破坏主要公共 API 且任务未要求：最高 60 分。
- 依赖外部网络、全局状态或本机特殊配置才能通过：最高 70 分。
- 大规模无关重构导致评审无法确认意图：最高 75 分。
- Agent 运行超时或崩溃：最高 50 分。

## 1. 功能正确性

1. 解决任务描述中的主要用户问题，而不是只满足测试样例。
2. 保持公开 API、CLI 参数、HTTP 路由、配置键等外部契约兼容。
3. 正确处理正常输入、空输入、缺省值和可选参数。
4. 正确处理非法输入，并返回清晰错误或异常。
5. 修复根因，而不是在测试数据上写特判。
6. 多步骤业务流程中的状态转换保持一致。
7. 并发、重复调用、重试时行为稳定。
8. 输出格式、排序、字段名、状态码与需求一致。
9. 原有成功路径不能被新逻辑破坏。
10. 与任务无关的行为不能发生可观察变化。

## 2. 隐藏测试与回归保护

1. 区分 fail-to-pass 测试和 pass-to-pass 测试。
2. fail-to-pass 必须证明原始缺陷被修复。
3. pass-to-pass 必须证明已有能力未回退。
4. 隐藏测试覆盖公开测试未覆盖的边界值。
5. 隐藏测试覆盖异常路径和错误恢复路径。
6. 隐藏测试覆盖真实项目中常见数据规模。
7. 隐藏测试覆盖不同运行顺序，避免测试间污染。
8. 隐藏测试验证没有修改测试、mock、fixture 或评分入口。
9. 隐藏测试在干净 worktree 中重复运行结果一致。
10. 回归测试失败时，即使新增功能通过，也不能给高分。

## 3. 需求遵循与边界覆盖

1. 完整覆盖任务说明中的每一条 success criteria。
2. 对含糊需求采用保守、局部、可解释的实现。
3. 不实现任务未要求的大功能。
4. 不改变用户未授权的数据模型或配置语义。
5. 对文档、设计稿、design.md、AGENTS.md 等约束逐条响应。
6. 对平台、语言版本、依赖版本限制保持兼容。
7. 对国际化、编码、路径分隔符等环境差异有处理。
8. 对边界值有明确行为，而不是依赖偶然实现。
9. 需求冲突时在结果中留下清晰说明或最小合理取舍。
10. 任务要求“只计划不改代码”时不得修改代码。

## 4. 代码质量与可读性

1. 命名表达业务含义，不使用含糊缩写。
2. 控制函数、类、模块的职责边界。
3. 删除重复逻辑或复用已有局部抽象。
4. 不引入不必要的新框架、新依赖或全局状态。
5. 条件分支清晰，复杂分支有必要注释。
6. 错误处理具体，不吞异常、不裸 except。
7. 类型、数据结构和返回值保持一致。
8. 代码格式符合项目已有风格。
9. 不留下调试打印、临时文件路径或 TODO 占位。
10. 实现能被代码评审快速理解和定位风险。

## 5. 架构集成与最小侵入

1. 优先使用项目已有工具函数、服务层、适配器和配置系统。
2. 修改范围与任务范围匹配。
3. 不跨层直接访问不该访问的内部实现。
4. 不绕开既有鉴权、校验、缓存、日志或事务边界。
5. 保持模块依赖方向合理，不引入循环依赖。
6. 保持 CLI、Web API、GUI、测试之间的数据契约一致。
7. 兼容现有历史数据、报告文件和配置文件。
8. 对兼容层、别名包、导出路径保持稳定。
9. 迁移或重命名文件时提供清晰过渡。
10. 新增抽象必须降低复杂度，而不是只包装一层。

## 6. 安全与权限边界

1. 不泄露 token、密钥、路径中的敏感信息或用户数据。
2. 不把不可信输入拼进 shell 命令。
3. 文件读写限制在任务工作区或明确授权目录。
4. 删除、移动、覆盖文件前必须验证目标路径。
5. Web API 对路径、任务文件、报告文件做范围校验。
6. 不默认开放危险权限；必须可配置、可审计。
7. 日志中避免记录敏感 prompt、凭据或完整私有数据。
8. 依赖执行外部工具时记录命令和退出码。
9. 不通过修改测试、评分器、环境变量来获得虚假高分。
10. 发现权限不足时明确失败，而不是静默跳过。

## 7. 性能与资源使用

1. 运行时间在任务规定 timeout 内留有余量。
2. 算法复杂度与输入规模匹配。
3. 不重复执行昂贵命令、网络请求或文件扫描。
4. 对大文件、大 diff、大日志有流式或分页处理。
5. 子进程能正确超时和清理进程树。
6. 临时目录、worktree、构建产物能被清理或忽略。
7. GUI 操作不阻塞主线程和关键交互。
8. 报告、排行榜、仪表盘生成能处理多次历史运行。
9. 不为小任务引入重型依赖或后台服务。
10. 性能优化不能牺牲正确性和可读性。

## 8. 测试质量

1. 新增或更新测试能在修复前失败、修复后通过。
2. 测试覆盖成功路径、失败路径和边界路径。
3. 测试断言具体行为，而不是只断言“不报错”。
4. 测试文件放在项目约定目录和命名规则下。
5. 测试不依赖运行顺序、当前机器、真实网络或脏状态。
6. 对 CLI、API、核心函数按风险选择合适层级测试。
7. 回归测试保留原缺陷的最小复现。
8. 对生成类任务，未补测试不得拿满测试质量分。
9. 对规划类任务，测试计划必须说明验证命令和覆盖点。
10. 测试日志足够诊断失败原因。

## 9. 可维护性与长期演进

1. 变更易于后续扩展，不把临时规则硬编码到核心路径。
2. 配置项有默认值、说明和兼容策略。
3. 公共函数和数据模型有稳定语义。
4. 新增错误码、失败原因、报告字段保持可读。
5. 代码路径能被单元测试或 smoke 测试覆盖。
6. 文档同步更新用户可见行为。
7. 不制造技术债，例如复制大段代码或隐藏分支。
8. 对平台差异有明确处理，例如 Windows 编码和路径。
9. 结果文件结构稳定，历史报告仍可读取。
10. 未来新增 agent、任务类型或评测维度时无需大改。

## 10. 交付过程与可审计性

1. 保存 agent stdout、stderr、测试日志、diff 和 summary。
2. 记录 agent 命令、退出码、耗时、是否超时。
3. 记录每个评分维度的原始证据和扣分原因。
4. 报告中展示硬性门槛触发情况。
5. 排行榜可按任务类型、agent、失败原因过滤。
6. 每次运行使用隔离 worktree，避免污染主工作区。
7. 评分器版本写入 result.json，便于历史对比。
8. 对人工评分项留下 reviewer、时间和备注。
9. 对自动评分项能从日志和 diff 复算。
10. 失败时保留足够上下文，避免只显示总分。

## 推荐自动化实现

第一阶段可以先自动化以下项目：

- 公开测试通过率。
- 隐藏测试通过率。
- fail-to-pass 与 pass-to-pass 分离统计。
- 是否修改测试或评分器。
- 是否新增或修改测试文件。
- diff 文件数、diff 行数、无关文件变更。
- forbidden files 与 forbidden patterns。
- agent 退出码、超时、运行时长。
- stdout/stderr 编码兼容和日志完整性。
- result.json 中记录评分器版本与评分明细。

## AgentArena Local 轻量 SWE-style 实现

AgentArena Local 不依赖 Docker、WSL2 或 SWE-bench 官方 harness。它保留本地 Git 仓库和 Git worktree 隔离，只吸收 SWE-bench 最核心的评测思想。

### task.yaml 字段

```yaml
baseline:
  commands:
    - name: bug-before
      command: pytest tests/test_bug.py::test_repro

fail_to_pass:
  commands:
    - name: bug-after
      command: pytest tests/test_bug.py::test_repro

pass_to_pass:
  commands:
    - name: regressions
      command: pytest tests/test_existing.py

hidden:
  commands:
    - name: hidden-edge-cases
      command: pytest tests_hidden
```

### 执行语义

- `baseline` 在 agent 修改前运行，用于确认任务确实能复现缺陷。
- `fail_to_pass` 在 agent 修改后运行，用于确认缺陷被修复。
- `pass_to_pass` 在 agent 修改后运行，用于确认旧功能没有回归。
- `hidden` 在 agent 修改后运行，用于降低只针对公开测试写特判的风险。
- 如果没有配置这些字段，系统继续使用旧的 `test` 或 `test_commands` 逻辑。

### resolved 判定

严格评测启用后，满足以下条件才算 resolved：

- `baseline` 没有通过，说明任务在原始状态下真实失败；未配置 baseline 时不阻塞。
- `fail_to_pass` 全部通过；未配置时视为无该项。
- `pass_to_pass` 全部通过；未配置时视为无该项。
- `hidden` 全部通过；未配置时视为无该项。

### 自动评分 breakdown

| 项目 | 分值 |
| --- | ---: |
| fail_to_pass | 30 |
| pass_to_pass | 15 |
| hidden | 15 |
| feature_completeness | 10 |
| constraints_passed | 8 |
| diff_quality | 7 |
| test_changes | 10 |
| time_efficiency | 3 |
| task_validity | 2 |
| violations | 扣分 |

硬性封顶：

- baseline 原本已经通过：最高 40。
- fail_to_pass 失败：最高 60。
- pass_to_pass 失败：最高 70。
- hidden 失败：最高 80。
- 未配置 hidden：最高 85。
- 未新增或修改测试：最高 80。
- 没有有效改动：最高 30。
- 未 resolved：最高 85。

第二阶段再加入：

- 静态分析。
- 复杂度变化。
- 依赖变化审计。
- 安全扫描。
- 性能基准。
- 人工复核表单。
- LLM 辅助评审，但不得作为唯一评分依据。
